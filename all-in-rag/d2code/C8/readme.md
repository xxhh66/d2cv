| 步骤                   | 操作                             | 为什么要这样设计？                                                                                                                                                                                                                                                |
| :--------------------- | :------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **第一步：文档加载**   | 递归查找 `.md` 文件              | **菜谱是分散的**（分目录存放），通过 `.rglob("*.md")` 可以一次性把所有菜谱文档自动召回，为后续构建知识库做准备。                                                                                                                                                  |
|                        | 加载 Markdown 文件，读取内容     | 菜谱的**结构化信息依赖标题层级**（如 `#`、`##`），保留原始 Markdown 格式才能准确识别结构。                                                                                                                                                                        |
|                        | 分配父文档 ID                    | 给每份菜谱一个**唯一且确定的ID（基于路径哈希）**，保证知识库可维护。如果文件移动或改名，ID会变化，我们就能自动感知到文档更新。                                                                                                                                    |
| **第二步：元数据增强** | 提取菜品分类（如川菜/粤菜）      | 解决 **“分类搜索”** 需求（如“推荐川菜”）。把“分类”提取成元数据，能实现精确过滤，比从文本中硬搜准确得多。                                                                                                                                                          |
|                        | 提取菜品名称                     | 解决 **“精确查找”** 需求（如“麻婆豆腐怎么做”）。将名称独立出来，可以优先匹配，避免检索时被内容干扰。                                                                                                                                                              |
|                        | 分析难度等级（如简单/中等/困难） | 满足 **“按难度筛选”** 需求（如“快速上手简单的菜”）。将难度元数据化，便于后续过滤。                                                                                                                                                                                |
| **第三步：结构分块**   | **一级/二级/三级标题分割**       | **关键一步！** 菜谱结构天然有层级： - `#` 菜名（最高层级） - `##` 分类/食材（第二层级） - `###` 步骤（第三层级）  **按标题分割**，能生成大小合适、语义独立的“段落块”。比如用户问“如何腌肉”，模型就能精准定位到“### 步骤1：腌制”这个块，而不必把整篇菜谱都塞进去。 |
|                        | 父子关系建立                     | 解决 **“分块后丢失层级”** 的问题。子块（如步骤）知道自己的父块（如菜名），这样检索时既能精准定位到细节，又能回溯到完整菜谱，兼顾“细节精准”和“上下文完整”。                                                                                                        |
| **第四步：后续处理**   | 输出文本块 chunks                | 将分割好的菜谱块喂给 Embedding 模型，构建向量索引。这些 **“既独立又带父子信息”** 的块，就是 RAG 高效检索的基础。                                                                                                                                                  |
|                        | 智能去重机制                     | 防止同一份菜谱因放在不同目录下而被重复加载，污染知识库，保证数据的纯净性。                                                                                                                                                                                        |
